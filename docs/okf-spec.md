# Memoss OKF Reference (v0.1)

Memoss vaults use **OKF** (Open Knowledge Format) â€” Markdown files with YAML frontmatter, compatible with [Google knowledge-catalog OKF Spec v0.1](https://github.com/GoogleCloudPlatform/knowledge-catalog/blob/main/okf/SPEC.md).

This document describes Memoss-specific conventions on top of the base spec.

---

## Document structure

Every OKF page is a `.md` file:

```markdown
---
type: Topic
title: Data Pipelines
description: How batch and streaming pipelines are modeled.
tags: [data, architecture]
resource: https://example.com/data-pipelines
---

# Data Pipelines

Body content in Markdown.

# Citations

- https://example.com/data-pipelines
```

### Required frontmatter

| Mode | Required fields | Use case |
|------|-----------------|----------|
| **Read** | `type` | Third-party bundles, partial imports |
| **Write** (agent) | `type`, `title`, `description` | Agent-authored pages |

Memoss validates in dual mode: permissive on read, strict on agent writes.

### Reserved filenames

| File | Purpose |
|------|---------|
| `index.md` | Directory listing (auto-generated by agents) |
| `log.md` | Vault activity log |

These files are skipped by the graph viewer and search indexing.

---

## Cross-links

Memoss uses **file-relative** links:

```markdown
See also [CQRS](../topics/cqrs-pattern.md).
```

Agents must write file-relative links only. The core `paths.ts` module resolves both file-relative and bundle-relative forms on read.

A typical research vault:

```

  viz.html              # generated by `memoss view`
```

## Page types (schema packs)

### Research pack

| Type | Directory | Purpose |
|------|-----------|---------|
| `Topic` | `topics/` | Core concept pages |
| `Reference` | `references/` | External source summaries |
| `Note` | `notes/` | Query answers and explorations |

### Personal pack

Same structure with `personal` defaults in `.memoss/instructions.md`.

## Agent write policies

When agents call `write_page`:

1. **Read-before-write** â€” must `read_page` first
2. **Frontmatter merge** â€” unknown keys preserved
3. **Body shrink guard** â€” warns if new body &lt; 30% of existing length
4. **Citation policy** â€” factual claims should include a `# Citations` section
5. **Draft branch** â€” when `git.draft_branch: true`, writes go to a draft branch until `memoss approve`
