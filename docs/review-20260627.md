## 一、架构设计层

### A1 — 模块边界与职责划分

**设计文档约定:** "Thin apps, fat core" — CLI 与 MCP 只做 I/O 编排，业务逻辑全部在 `@memoss/core`。

**实现对照:**

| 层面 | 评价 |
|------|------|
| **三层边界** | ✅ core/cli/mcp 三层边界清晰。cli 和 mcp 均只依赖 core，core 无内部包依赖。ESLint `depConstraints` 强制执行了此规则 (`eslint.config.mjs:23-36`)。 |
| **core 内部模块** | ✅ 模块职责划分合理：`okf/`(文档模型)、`engine/`(Agent 运行器)、`tools/`(Agent 工具)、`policies/`(安全策略)、`adapters/`(I/O 抽象)、`config/`(配置)、`lint/`(确定性检查)、`skills/`(提取技能)、`provenance/`(溯源)、`viewer/`(可视化)。每个模块有明确的单一职责。 |
| **Public API 控制** | ✅ `index.ts` 导出约 179 个符号，粒度适中。内部模块（policies 内部实现、prompts 等）不直接暴露。 |
| **潜在问题** | ⚠️ `engine/index.ts` 既导出 runner 函数又导出 runner-setup 的工具函数，边界略显模糊。`createRunnerSetup` 被同时从 `engine/index.ts` (line 168) 和专用路径导出，存在重复导出。 |

**具体发现:**
- `packages/core/src/index.ts` 第 168 行: `export { createRunnerSetup } from './engine/index.js';` — 该函数在 engine/index.ts 内已经被 barrel 导出过一次，core 的 index.ts 又重复导出，虽然不会导致运行时错误，但反映了 barrel export 管理不够严格。
- `tools/context.ts` 定义了 `ToolContext` 接口，但未被 `index.ts` 直接导出（通过 ToolRegistry 间接暴露），这可能是有意为之以减少 API 表面积。

**评分: 9/10**
> 三层架构贯彻到位，内部模块高内聚低耦合。barrel export 有少量冗余但不影响正确性。

### A2 — 扩展性与插件化设计

**三个扩展维度评估:**

#### 2a. Source Adapter 注册机制

✅ **设计良好。** `adapters/source-registry.ts` 采用工厂模式，`createSourceAdapter({ kind, uri })` 根据 `kind` 分发到 `source-file/source-web/source-github`。`SourceAdapter` 接口 (`adapters/types.ts`) 定义了清晰的契约: `listItems()` + `readItem(id)`。新增 adapter 只需实现该接口并注册即可。

#### 2b. Extraction Skills 系统

✅ **设计优秀。** `skills/` 模块实现了 skills.sh 兼容的插件化提取框架：
- `discovery.ts` — 发现 vault/user 级别的 SKILL.md
- `catalog.ts` — 构建 skill 目录供 Agent 选用
- `router.ts` — 路由决策（显式指定 > config 映射 > Agent 自选）
- `parse-skill-md.ts` — 解析标准 SKILL.md 格式
- `glob-match.ts` — URL 模式匹配
- `trust.ts` — Skill 信任验证

`ExtractKind` 类型 (skills/types.ts:19-28) 已覆盖 web/pdf/github/markdown/text/audio/video/unknown，扩展性好。

#### 2c. Schema Packs

✅ **设计简洁有效。** `schema-pack/` 模块提供 `personal/research/data-catalog` 三种模板，`switch-pack.ts` 支持运行时切换。模板是最小 token 替换（非 Handlebars），避免复杂模板引擎。

**评分: 9/10**
> 三个扩展点设计均遵循开闭原则。Skills 系统与 skills.sh 生态对齐是亮点。Schema Pack 可考虑未来支持社区自定义 pack 注册。

### A3 — 数据模型设计

**OKF 文档格式:**

```typescript
interface OKFDocument {
  frontmatter: OKFFrontmatter;  // type, title, description, resource, tags, sources, ...
  body: string;
}
```

**评估:**

| 方面 | 评价 |
|------|------|
| **类型设计** | ✅ `OKFFrontmatter` 使用索引签名 `[key: string]: unknown` 保留未知字段，这是有意的设计选择以支持 round-trip 保真（文档 §5.2）。 |
| **双模式校验** | ✅ `validateForRead` (仅 `type`) vs `validateForWrite` (`type`+`title`+`description`) — 符合设计文档约定。`isReservedFilename` 豁免 index.md/log.md 不参与严格写校验，逻辑正确。 |
| **Concept ID 体系** | ✅ `ConceptId = readonly string[]` 用路径段表示，`conceptIdToPath`/`pathToConceptId` 双向转换干净。段名正则 `^[A-Za-z0-9_][A-Za-z0-9_.-]*$` 限定严格。 |
| **链接格式** | ✅ 采用文件相对路径 (`../topics/other.md`)，兼容 GitHub/Obsidian/本地 FS。`resolveLink()` 正确处理相对/绝对/锚点剥离。 |
| **自定义解析器** | ✅ 自研 OKF parser (`okf/document.ts`) 实现了与 knowledge-catalog 对齐的语义：unterminated `---` 报错、frontmatter 非 mapping 报错、YAML 解析失败报错。使用 `yaml` (eemeli) 包的 `sortMapEntries: false` 保证 round-trip 稳定性。 |
| **Index/Log 设计** | ✅ `IndexDocument` 无 frontmatter（plain markdown），`LogEntry` 按日期分组。`FsKnowledgeStore.appendLog()` 的日期分组插入逻辑 (`fs-store.ts:37-62`) 正确处理了追加到已有日期的场景。 |

**具体发现:**
- `OKFFrontmatter.type` 标记为可选 (`type?: string`)，但 validator 要求 `type` 在两种模式下必填。这在类型层面无法表达"parse 时可选，validate 后必填"的语义。建议考虑区分 `ParsedOKFDocument`(全部可选) 和 `ValidatedOKFDocument`(必填字段保证)。
- `serializeOKF` (document.ts:49-61) 确保 body 以 `\n` 结尾，并在 frontmatter 和 body 之间插入空行，格式一致性良好。

**评分: 8.5/10**
> OKF 模型设计成熟，与 knowledge-catalog 参考实现对齐。类型系统未完美表达"解析后 vs 校验后"的状态差异。

### A4 — 依赖方向与耦合

**依赖图:**

```
@memoss/cli ──────► @memoss/core ◄────── @memoss/mcp-server
    │                      │
    └── @memoss/mcp-server (workspace:^)
```

**评估:**

| 检查项 | 结果 |
|--------|------|
| **包间依赖** | ✅ 单向无环。core 是唯一下游依赖方向（cli→core, mcp→core）。 |
| **ESLint 边界** | ✅ `depConstraints` 强制 `scope:cli` 和 `scope:mcp` 只能依赖 `scope:core`，`scope:core` 无内部依赖。 |
| **CLI → MCP** | ⚠️ `apps/cli/package.json` 声明了 `@memoss/mcp-server: workspace:^` 依赖。CLI 通过 `memoss serve` 启动 MCP server 时需要它。这违反了设计文档中 CLI 只依赖 core 的声明。不过这是合理的架构选择——CLI 作为入口需要集成 MCP server。ESLint 标签实际允许这个依赖，因为 mcp-server 有 `scope:mcp` 标签而 CLI 只有 `scope:cli` 标签但 `depConstraints` 只限制了 `scope:cli` → `scope:core`。**这里其实存在不一致：ESLint 规则说 CLI 只能依赖 scope:core，但 CLI 实际依赖了 mcp-server。** |
| **core 外部依赖** | ✅ 仅依赖 AI SDK (`ai`, `@ai-sdk/anthropic`, `@ai-sdk/openai`)、文件系统 (`fast-glob`)、Git (`simple-git`)、Web 解析 (`linkedom`, `readability`, `turndown`)、YAML (`yaml`)、Schema (`zod`)、GitHub API (`@octokit/rest`)。无重量级框架。 |
| **内部模块耦合** | ✅ `ToolContext` 接口 (`tools/context.ts`) 作为工具层和引擎层的依赖反转边界。`PolicyRunner` 封装了策略组件。Runner 通过 `createRunnerSetup()` 统一装配依赖。 |
| **循环依赖** | ✅ 未发现。import 方向均为单向：engine → tools/policies/adapters, tools → adapters/policies/okf。 |

**具体发现 —— CLI → MCP 依赖不一致:**

`eslint.config.mjs` line 26-28 声明 `scope:cli` 只能依赖 `scope:core`，但 `apps/cli/package.json` 实际依赖了 `@memoss/mcp-server` (scope:mcp)。这可能因为:
1. ESLint `depConstraints` 仅检查 `onlyDependOnLibsWithTags` 的 **lib** 类型 tag，mcp-server 也是 lib 所以规则设计上可能有问题
2. 需要检查 ESLint 是否实际报告了此违规

**评分: 7.5/10**
> 整体依赖方向干净。CLI→MCP 依赖与文档设计不一致（或 ESLint 规则未正确覆盖此场景），需要核实。

### B1 — Runner 架构设计

**现有 Runner:**

| Runner | 文件 | 用途 |
|--------|------|------|
| `ingest-runner.ts` | Ingest Agent | 知识入库 |
| `query-runner.ts` | Query Agent | 知识查询 |
| `lint-runner.ts` | Lint Agent | 质量检查 |
| `extract-runner.ts` | Extract Agent | 提取预处理 |
| `validate-runner.ts` | Validation | 内容验证 |
| `tuning-runner.ts` | Tuning | 质量调优 |

**评估:**

| 方面 | 评价 |
|------|------|
| **拆分粒度** | ✅ 每个 Runner 职责单一，符合设计文档 §7.4 的工具子集策略。`ingest-runner` 编排了 validate → tuning → ingest 的完整流水线，实现了设计文档 M4 的流程。 |
| **公共逻辑复用** | ✅ `runner-setup.ts` 提供了 `createRunnerSetup`/`ensureDraftBranch`/`resolveRunnerModel` 等公共函数。`orchestrator.ts` 提供了 `runAgentLoop`/`runAgentStream` 两种模式。 |
| **状态管理** | ✅ Runner 本身无状态，状态集中在 `ToolContext`（通过 `PolicyRunner` 跟踪 read/write 路径）。`PolicyRunner.reset()` 在每次 ingest 前调用。 |
| **错误处理** | ⚠️ `runAgentLoop` 中 `finishReason === 'stop'` 以外都返回 `incomplete`，但没有对 `finishReason === 'error'` 做特殊处理。AI SDK 的 `finishReason` 包括 `'stop' | 'length' | 'content-filter' | 'tool-calls' | 'error' | 'other' | 'unknown'`，其中 `'error'` 和 `'content-filter'` 需要区别对待。 |

**具体发现 —— Ingest Runner 流水线嵌套:**
`ingest-runner.ts` line 53-190 展示了 validate → tuning → ingest 的串行编排：
1. `resolveIngestSource()` — 解析提取来源
2. `runValidate()` (unless `--skip-validate`) — 内容质量验证
3. `runTuningPass()` (unless `--skip-tuning`) — 生成质量 overlay
4. `createRunnerSetup()` + `runAgentLoop()` — 主 ingest 循环

这个流水线设计合理，但 `runValidate` 失败返回 `status: 'rejected'` 的结果时，`formatValidationFailure()` 生成的文本比较简略（仅列出 issues）。对用户来说，知道"为什么被拒绝"和"如何改进"同样重要。

**评分: 8/10**
> Runner 拆分清晰，公共逻辑复用良好。Agent 循环的 finishReason 处理不够细粒度。

### B2 — Prompt 工程与系统指令

**Prompt 加载机制** (`engine/prompts/load.ts`):

- 模板文件位于 `engine/prompts/*.md`，通过 `fileURLToPath(import.meta.url)` 定位
- 支持缓存 (`Map<string, string>`)
- 变量替换：`{{vault_name}}`, `{{schema_pack}}`, `{{instructions}}`, `{{date}}`, `{{quality_patterns}}`, `{{schema_overlay}}` + 自定义 extra 变量

**系统 Prompt 组装** (`engine/context.ts`):

```typescript
buildSystemPrompt({
  prompt: 'ingest',           // 选择模板
  vaultName, schemaPack,      // 基础上下文
  instructions, date,          // vault 指令 + 日期
  qualityPatterns,             // 通用质量模式
  schemaOverlay,              // schema pack 专属覆盖
  extra: { quality_overlay, save_instructions, fix_instructions, ... }
})
```

**评估:**

| 方面 | 评价 |
|------|------|
| **模板管理** | ✅ Markdown 模板文件化管理，与代码解耦。`renderTemplate` 使用简单正则 `\{\{(\w+)\}\}` 而非复杂模板引擎。 |
| **上下文分层** | ✅ 四层结构：基础模板 → 质量模式 → schema overlay → 运行时注入。设计合理。 |
| **质量模式** | ✅ `loadQualityPatterns()` 加载通用质量指引，所有 prompt 共享。 |
| **潜在问题** | ⚠️ Prompt 模板内容未在此次评审中查看（`.md` 文件不在已读范围内）。模板质量直接影响 Agent 行为，建议纳入评审范围。 |
| **Token 控制** | ⚠️ 未发现显式的 token 预算/截断机制。系统 prompt + vault instructions + quality patterns + schema overlay 累加后可能较长。对于大型 vault 或长 instructions.md 需要关注。 |

**评分: 7.5/10**
> Prompt 组装架构合理，分层清晰。Prompt 模板内容质量和建议的 token 控制机制需进一步评估。

### B3 — Tool 定义与工具编排

**Tool 定义** (`tools/define-tool.ts`):

```typescript
export function defineTool<T extends z.ZodTypeAny>(
  ctx: ToolContext,
  definition: ToolDefinition<T>,
): Tool {
  return tool({
    description: definition.description,
    inputSchema: definition.inputSchema,     // Zod schema → AI SDK
    execute: async (args) => definition.execute(args as z.infer<T>, ctx),
  });
}
```

**Tool 子集** (`engine/pick-tools.ts`):

| 模式 | 工具集 |
|------|--------|
| Ingest | read_page, write_page, list_pages, read_index, write_index, read_log, append_log, fetch_url, read_source, list_sources, git_commit, git_diff, git_create_branch |
| Query | read_page, read_index, search_kb |
| Query + Save | + write_page, append_log |
| Lint | read_page, list_pages, read_index, search_kb |
| Lint + Fix | + write_page, write_index, append_log, git_commit, git_diff, git_create_branch |

**评估:**

| 方面 | 评价 |
|------|------|
| **Zod Schema 集成** | ✅ `defineTool` 将 Zod schema 传递给 AI SDK `tool()` 的 `inputSchema`，类型安全。17 个工具的 Zod schemas 集中在 `tools/tool-schemas.js`，风格统一。 |
| **Tool 执行链路** | ✅ `write_page` 执行链路完整：(1) assertReadFirst → (2) mergeFrontmatter → (3) checkHeadingsPreserved → (4) checkResourceUnchanged → (5) checkBodyNotShrunk → (6) validateForWrite → (7) citation check → (8) writePage。符合设计文档 §5.6。 |
| **Policy 可配置** | ✅ 每个 policy check 的 action (`error`/`warn`/`off`) 可通过 `PoliciesConfig` 调整。`enforcePolicyViolation` 对 `error` 级别抛出异常阻断。 |
| **动态 import** | ⚠️ `createFetchUrlTool` (page-tools.ts:196-211) 和 `createSearchKbTool` (page-tools.ts:183-193) 使用动态 `import()` 来延迟加载。这可能是为了避免循环依赖或减少启动开销。但 fetchUrl 和 searchKb 并非大模块，动态 import 的必要性需要确认。 |
| **工具间协作** | ✅ `read_page` 自动调用 `policies.augment.markRead()` 追踪已读页面，为 `write_page` 的 `assertReadFirst` 提供保障。这是精妙的隐式协作设计。 |
| **MCP 工具映射** | ✅ `getToolInputSchema()` (extract-schema.ts) 从 AI SDK Tool 中提取 Zod schema 供 MCP 生成 `inputSchema`，避免重复定义。 |

**评分: 8.5/10**
> Tool 层设计成熟：Zod 类型安全、可配置 policy 链、隐式协作。动态 import 的使用模式需要统一决策。

### B4 — 安全策略与护栏

**策略系统** (`policies/`):

| 策略 | 文件 | 机制 |
|------|------|------|
| **Read-before-Write** | augment.ts:39-50 | `augment.markRead()` 追踪 → `assertReadFirst()` 检查。`require_read_before_write: true` (默认) |

| **Resource Unchanged** | augment.ts:96-116 | 禁止修改 `resource` 字段。`preserve_resource_field: 'error'` (默认) |
| **Citation** | citation.ts:6-41 | 检测 `# Citations` 段落 + 实质性内容 ≥ 80 字符。`require_section: 'warn'` (默认) |
| **Reference Mint** | reference-mint.ts | 四条件检查：slug 不在阻塞列表、title 不是元描述词、body ≥ 80字符。`gate_mode: 'warn'` (默认) |
| **Path Traversal** | fs-store.ts:20-31 | `resolveVaultPath()` 拒绝 `..` 和绝对路径逃逸 |

**评估:**

| 方面 | 评价 |
|------|------|
| **护栏完整性** | ✅ 覆盖了写入安全的主要维度。read-before-write 是核心信任原语。 |
| **可配置性** | ✅ 所有策略的 action 三级可配 (`error/warn/off`)，`policiesConfigSchema` (config.ts:21-66) 定义清晰。 |
| **Citation 检测** | ⚠️ `citation.ts` 的启发式比较简单：只检查是否存在 `# Citations` 标题 + 是否有 ≥80 字符的非标题行。无法区分真正的实质性声明和代码块/引用块。可能产生误报（技术文档中的代码段）或漏报（短声明但有实质内容）。 |
| **Reference Mint** | ✅ 四条件 minting 直接移植自 knowledge-catalog reference agent 的验证规则，与参考实现一致。slug blocklist 包含常见元页面词。 |
| **路径安全** | ✅ `FsKnowledgeStore.resolveVaultPath` 使用两层防护：先检查 `../` 字符串，再验证规范化路径在 vault root 内。但要注意 Windows 上 `..\\` 也需要处理（已在 `normalizeRelativePath` 中统一为正斜杠）。 |

**评分: 8/10**
> 多层策略护栏设计成熟，可配置性好。Citation 检测启发式可进一步细化。

### D1 — Draft-First 信任模型

**实现:**

```typescript
// runner-setup.ts:56-71
async function ensureDraftBranch(ctx, operation): Promise<string | undefined> {
  if (!ctx.config.git.enabled || !ctx.draftMode) return undefined;
  if (!await ctx.git.isRepo()) await ctx.git.init();
  const branch = createDraftBranchName(operation); // memoss/draft/<op>-<timestamp>
  await ctx.git.createBranch(branch);
  return branch;
}
```

**审批/拒绝流程:**
1. Agent 在 draft branch 上执行操作
2. 用户运行 `memoss approve` → `git checkout main` → `git merge --ff-only` → `git branch -d`
3. 用户运行 `memoss reject` → 丢弃 draft branch

**评估:**

| 方面 | 评价 |
|------|------|
| **默认安全** | ✅ `draft_branch: true` 默认开启。Agent 写入不直接污染 main。 |
| **分支命名** | ✅ `memoss/draft/<operation>-<YYYYMMDD-HHmmss>` 格式包含操作类型和时间戳，可审计。 |
| **`--no-draft` 警告** | ✅ 需要显式 flag 才能跳过 draft（设计文档 §14）。实现中 `opts.noDraft` 控制 `useDraft` 标志。 |
| **冲突处理** | ⚠️ `git merge --ff-only` 失败时 CLI 提示手动解决，但没有提供合并冲突的辅助工具。对于多用户场景（Phase 2），这需要强化。 |
| **非 Git 模式** | ✅ `git.enabled: false` 时跳过所有 git 操作并警告。 |
| **reject 安全性** | ⚠️ `reject` 丢弃 draft branch，未提交的变更会永久丢失。没有 `git stash` 或备份机制。 |

**评分: 8/10**
> Draft-first 是核心信任原语，实现简洁有效。冲突处理和 reject 安全性有提升空间。

### E3 — 性能与资源消耗

**LLM 调用分析:**

| 操作 | LLM 调用次数 | 典型 Token 消耗 |
|------|------------|---------------|
| Ingest | 1-3 次 (validate + tuning + ingest agent loop) | 高（默认模型 Sonnet, max 50 steps） |
| Query | 1 次 (query agent) | 中（轻量模型 Haiku） |
| Lint | 1 次 (lint agent) + 确定性检查 | 中低 |
| Extract | 1 次 (extract agent, max 15 steps) | 中 |

**文件系统 I/O:**

| 操作 | 模式 | 规模 |
|------|------|------|
| `search_kb` | 全文件扫描 (`fast-glob` + 逐行 grep) | O(n * m) — n 页数 × m 平均行数 |
| `list_pages` | `fast-glob` 枚举 | O(n) |
| `runDeterministicLint` | 全量读取 + 链接分析 | O(n) |

**评估:**

| 方面 | 评价 |
|------|------|
| **LLM 成本** | ⚠️ Ingest 串行执行 validate + tuning + main loop，每次最多 50 steps。对于一个 URL 的 ingest 可能消耗数千 token。没有 token 消耗统计或预算控制。 |
| **Search 扩展性** | ⚠️ Phase 1 使用纯 Node.js grep（逐文件逐行扫描）。设计文档提到 ~200 页阈值是合理的，但当前实现没有分页或并行。对于 500+ 页 vault，每次 `search_kb` 会全量扫描。 |
| **Lint 扩展性** | ⚠️ `runDeterministicLint` 对所有页面执行全量读取和链接分析。对于大型 vault (>500 页) 可能较慢。 |
| **HTML 截断** | ✅ `fetchUrl` 截断至 40KB UTF-8 (与 knowledge-catalog 一致)。防止单次 fetch 消耗过多 token。 |
| **缓存** | ✅ prompt 模板有内存缓存。extraction 有文件缓存 (`extract-cache.ts`)。但 lint 结果没有缓存。 |
| **并行** | ⚠️ `searchKb` 对文件逐个顺序处理。可以并行扫描。 |

**具体性能关注点:**
1. `search_kb.ts` line 35-72: 对每个文件执行 `readFileSync` + 逐行 `toLowerCase().includes()`。对于大型 vault，可考虑分页返回或限制结果数量。
2. `runDeterministicLint` (`checks.ts` line 24-105): 对每个页面执行两次读取（readPage + readIndex），且全量扫描链接。建议添加进度回调。
3. Ingest 流水线的 LLM 调用次数固定（validate + tuning + main），没有根据源复杂度自适应调整。

**评分: 6.5/10**
> 小规模 vault (<200页) 性能可接受。大型 vault 和 LLM 成本控制在当前实现中未充分优化。
